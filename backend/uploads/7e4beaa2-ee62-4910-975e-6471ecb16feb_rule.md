# Git Commit 规范文档

## 概述

本文档定义了项目中的 Git Commit 提交规范，旨在提高代码历史的可读性、可维护性和自动化版本管理能力。

## Commit 消息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 格式说明

- `<type>`: 提交类型（必需）
- `<scope>`: 影响范围（可选）
- `<subject>`: 简短描述（必需）
- `<body>`: 详细描述（可选）
- `<footer>`: 脚注信息（可选）

## Type 类型

| 类型       | 说明                           | 示例                         |
| ---------- | ------------------------------ | ---------------------------- |
| `feat`     | 新功能                         | `feat: 添加用户登录功能`     |
| `fix`      | Bug 修复                       | `fix: 修复登录超时问题`      |
| `docs`     | 文档更新                       | `docs: 更新 API 文档`        |
| `style`    | 代码格式（不影响功能）         | `style: 统一缩进格式`        |
| `refactor` | 重构（既不是新功能也不是修复） | `refactor: 优化用户验证逻辑` |
| `perf`     | 性能优化                       | `perf: 优化图片加载速度`     |
| `test`     | 测试相关                       | `test: 添加单元测试`         |
| `chore`    | 构建/工具/依赖更新             | `chore: 升级依赖包版本`      |
| `ci`       | CI/CD 配置                     | `ci: 更新部署流程`           |
| `revert`   | 回退提交                       | `revert: 回退至 v1.0.0`      |

## Scope 范围

根据项目模块定义，常见范围示例：

- `api` - API 接口
- `ui` - 界面组件
- `db` - 数据库相关
- `config` - 配置文件
- `auth` - 认证授权
- `utils` - 工具函数
- `docs` - 文档

## Subject 规范

- 使用祈使句，现在时态（"添加" 而非 "添加了"）
- 首字母小写
- 不以句号结尾
- 长度不超过 50 个字符
- 清晰描述本次提交做了什么

### ✅ 正确示例

```
feat: 添加用户注册接口
fix: 解决分页查询偏移量计算错误
docs: 补充环境配置说明
```

### ❌ 错误示例

```
feat：添加新功能        # 使用了中文冒号
更新代码                # 缺少 type
fix: 修复了一个bug。    # 以句号结尾，描述不清晰
```

## Body 规范

- 详细说明"为什么"做此修改
- 说明与之前行为的差异
- 每行不超过 72 字符
- 可选，复杂变更建议添加

### 示例

```
feat: 添加 JWT 认证中间件

实现基于 JWT 的请求认证机制，包括：
- Token 生成和验证
- Token 过期处理
- 白名单支持

该改动解决了无状态认证的需求，提升了系统安全性。
```

## Footer 规范

### 关联 Issue

```
feat: 添加用户权限管理

Closes #123, #124
```

### Breaking Changes

```
refactor: 重构用户服务接口

BREAKING CHANGE: 用户服务 API 从 RESTful 变更为 GraphQL，
客户端需要进行相应调整。
```

## 完整示例

### 示例 1：新功能提交

```
feat(auth): 实现双因素认证功能

添加基于 TOTP 的双因素认证，增强账户安全性：
- 支持 Google Authenticator
- 支持备用恢复码
- 提供绑定和解绑接口

Closes #234
```

### 示例 2：Bug 修复提交

```
fix(api): 修复并发请求下 token 刷新异常

在 token 刷新过程中，如果同时有多个请求到达，
会导致 token 被重复刷新的问题。

解决方案：添加刷新锁机制，确保同一时间只有一个刷新操作。

Closes #456
```

### 示例 3：文档提交

```
docs(readme): 补充 Docker 部署说明

添加使用 Docker Compose 快速启动项目的步骤，
包括环境变量配置和数据卷挂载说明。
```

## 工具推荐

### 1. Commitizen

交互式提交工具，确保规范格式：

```bash
# 安装
npm install -g commitizen

# 使用
git cz
```

### 2. Commitlint

检查提交消息是否符合规范：

```bash
# 安装
npm install --save-dev @commitlint/cli @commitlint/config-conventional

# 配置
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

### 3. Husky + lint-staged

Git Hooks 自动化检查：

```json
{
  "husky": {
    "hooks": {
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS"
    }
  }
}
```

## 版本管理建议

### 语义化版本

基于提交类型自动生成版本号：

- **Major**（主版本）：包含 `BREAKING CHANGE`
- **Minor**（次版本）：包含 `feat` 类型
- **Patch**（补丁版本）：包含 `fix`、`perf` 等类型

### 生成 CHANGELOG

使用 standard-version 或 semantic-release 自动生成变更日志：

```bash
npx standard-version
```

## 团队约定

1. 每个提交只做一件事，保持原子性
2. 提交前确保代码通过所有测试
3. 禁止提交包含调试代码或临时文件
4. 提交消息使用中/英文均可，但团队内保持一致
5. 重要功能或破坏性变更必须在 body 中详细说明
